Field structure

Each entry in the fields array is an object with two required keys (key and type) and the rest optional. The system infers reasonable values when not explicitly declared.

key required
Unique identifier for the field at its nesting level. It maps directly to the key in the resulting model. In groups, each level has its own namespace, so the same key can appear in different groups without conflict.
type required
Controls which UI control is rendered and how the value is serialized. See the Field types tab for the complete list.
dataType
string · number · boolean. Inferred from type when omitted (checkbox → boolean, number → number, others → string). Only needs to be declared explicitly when the visual type does not match the desired data type.
label
Visible text accompanying the control in the UI. If omitted, the key value is displayed auto-formatted (camelCase → separated words).
defaultValue
Initial value of the field when creating or resetting the form. Must be compatible with dataType. If omitted, text fields default to "", numeric to 0, and boolean to false.
props
Additional control options:
  • placeholder — ghost text inside the input while empty
  • hint — help text that appears below the field
  • description — descriptive paragraph that appears above the field
  • options — [{{ '{' }} value, label {{ '}' }}] array required for select
hidden · disabled · readonly
Reactive conditions re-evaluated on every model change. Accepts {{ '{' }} "expr": "..." {{ '}' }} for self-contained DSL expressions, or {{ '{' }} "fn": "key" {{ '}' }} to call a function registered in provideJsonForms(). See the DSL and Registered functions tabs.
validators · asyncValidators
Arrays of validation rules. Synchronous ones are evaluated on every value change and block submission on failure. Async ones support debounce to reduce server calls. See the Validators tab.
fields group only
Array of nested fields that make up the group. In the model, the group produces an object whose key is the parent field key and whose value contains the inner fields.
item array only
Definition of the repeatable element. The UI automatically generates "Add" and "Remove" buttons. The model stores an array of objects following the structure of item. Typically item is a group with its own fields.

The type simultaneously determines the visual control that is rendered and the data type the field produces in the model. Primitive types produce a scalar value; composite types (group, array) produce nested structures that are reflected directly in the form model.

Validators (validators)

Validators are declared as an array under the field's validators key. All support the optional "message" key to customize the error text. There are four categories based on their origin and execution mode: standard, DSL, registered synchronous, and asynchronous.

Standard · self-contained
Rules built into the engine that require no external registry: declared solely in the JSON. These are required (mandatory field), email (email format), min / max (numeric range), minLength / maxLength (text length), and pattern (regular expression). All support "message" to override the default error.
expr · cross-field validation with DSL
The field is valid when the DSL expression evaluates to true. Useful for comparing the value against another form field, for example "value === model.emailConfirm". Runs in the same safe DSL evaluator (no eval), so it is also self-contained. See the Expression DSL tab for available syntax.
fn · registered synchronous function
Reference to a function registered in provideJsonForms().validators. Receives a context with value() (current value), valueAt(key) (value of another field), and model() (full model snapshot). Must return undefined if the field is valid, or {{ '{' }} kind, message {{ '}' }} if not. Ideal for complex logic that does not fit in the DSL: Angular services, advanced regex, etc.
Async · asynchronous validation
Declared in the field's asyncValidators array (not validators). References a key registered in provideJsonForms().asyncValidators. The "debounce" field (ms) prevents excessive server calls while the user is typing. The pattern uses Angular's resource: params defines what to send, factory creates the resource, onSuccess interprets the result, and onError handles network failures. 
{{ ref.validators }}

Groups and arrays

In addition to primitive types, you can compose complex data structures with group (nested object) and array (repeatable list of elements). Both can be combined and nested without depth limit.

group — nested object

Groups related fields under a common key. The group's value in the model is an object containing the values of its internal fields. For example, an "address" group with "street" and "city" fields produces {{ '{' }} address: {{ '{' }} street: "...", city: "..." {{ '}' }} {{ '}' }} in the model.

Groups can have their own label, hidden, disabled, and validators, which affect the entire block. A hidden group also hides all its inner fields.

array — repeatable list

Generates a list of zero or more entries where the user can add or remove rows. Each row follows the item definition, which is usually a group with its own fields. The UI adds "Add" and "Remove" buttons automatically.

The model stores an array of objects: if item.fields has "name" and "phone", the result is [{{ '{' }} name: "...", phone: "..." {{ '}' }}, ...]. The item key does not appear in the model; only the internal fields keys are stored directly at each array position.

Tip: arrays can contain groups that in turn contain other arrays. Validation and reactive conditions (hidden, disabled) work at any nesting level.

Expression DSL

The DSL is a JavaScript-like expression evaluator built into the engine. Used in hidden, disabled, readonly, and in expr validators. It does not use eval: the syntax tree is parsed once when the definition loads and re-evaluated reactively with signals whenever the model changes, with no additional cost.

Registered functions and validators

When logic does not fit in the DSL — closures over Angular services, HTTP calls, complex calculations, access to injectables — it is registered in TypeScript via provideJsonForms() in app.config.ts and referenced from the JSON by text key. There are three independent registries.

functions — reactive conditions
Functions that control hidden, disabled, and readonly. Receive a context with:
  • value() — signal with the current field value
  • valueAt(key) — signal with the value of another field by key
  • model() — signal with the full model snapshot
Must return true to activate the condition or false to deactivate it. Referenced with {{ '{' }} "fn": "keyName" {{ '}' }} in the JSON. Re-evaluated reactively with Angular signals whenever the model changes.
validators — synchronous validation
Custom synchronous validators. Receive the same context as functions. Must return undefined if the field is valid, or an object {{ '{' }} kind: string, message: string {{ '}' }} if not. The kind identifies the error type (useful for conditional styles) and message is the text the user sees. Referenced with {{ '{' }} "kind": "fn", "fn": "key" {{ '}' }} in the validators array.
asyncValidators — asynchronous validation
Asynchronous validators based on Angular's resource primitive. Each entry defines four properties:
  • params — function that extracts the value to send (e.g. the field text)
  • factory — creates the resource with the async loader (fetch, HttpClient…)
  • onSuccess — interprets the result and returns an error or undefined
  • onError — handles network or server failures and returns a fallback error
In the JSON they are used in the asyncValidators array with {{ '{' }} "kind": "key", "debounce": 400 {{ '}' }}. The debounce delays the request while the user is still typing, avoiding unnecessary calls.

System architecture

How signal-jsonforms translates JSON into reactive forms: the four engine layers, the compilation pipeline, and the UI-agnostic render system.

@switch (archTab()) { @case ('overview') {

Overview

Signal Forms is designed for compile-time typed structures: the model and its paths are known before running the app. signal-jsonforms inverts that flow: the form shape arrives as JSON at runtime. The library acts as an interpreter — it builds the model and the schema function dynamically — without reimplementing anything Signal Forms already provides.

Data-driven
The form's structure, validation, and behavior are declared in JSON. Modifying a form does not require touching TypeScript: just update the definition.
UI-layer agnostic
The core has no dependencies on Material, PrimeNG, or any kit. Integration is done via FieldTypeRegistry: changing the UI kit means only changing the registry, without touching the engine.
Faithful to Signal Forms
State, reactivity, and validation are managed by Signal Forms. The library is an interpreter that translates JSON → form(), not a reimplementation: it always delegates to the platform and leverages its native reactivity.
Isolated from experimental risk
@angular/forms/signals is marked @experimental and may change between versions. Every call to its API lives behind a single adapter layer: if the API breaks, it is fixed in one module.
Core tension: Signal Forms uses statically typed paths (path.address.city). At runtime, the library resolves them dynamically with path["address"]["city"] — identical in JavaScript — without static typing inside the engine. Typing is optionally recovered at the public boundary via generics.

System layers

Four layers with clear boundaries. The golden rule: only the adapter layer imports @angular/forms/signals. The rest works against the library's internal interfaces. 

{{ arch.layers }}
Consumer (app)
Defines the JSON form configuration and, optionally, registers in provideJsonForms() the field components, synchronous and asynchronous validators, and functions for advanced conditional logic.
Render layer
<jf-form> acts as the host. FieldRenderer traverses the FieldNode[] recursively and dynamically instantiates the component associated with each type, passing it the FieldTree node and the field configuration.
Engine layer
The pure core: no UI or Signal Forms dependencies. Comprises the Parser (JSON → IR), the ModelBuilder (IR → signal), the SchemaCompiler (IR → schemaFn), and the expression, function, and validator registries.
Adapter layer
The only module that imports @angular/forms/signals. Wraps form(), validate(), validateAsync(), applyWhen(), applyEach(), and all standard validators behind the library's own interfaces.
} @case ('engine') {

Engine: from JSON to form()

The engine transforms the JSON definition into an active Signal Forms form through four chained steps. The result is a (model signal, FieldTree) pair that the renderer consumes and that Signal Forms keeps reactive.

1 · Parser / Normalizer
Validates the JSON against a meta-schema (Zod) and transforms it into an array of FieldNode with calculated absolute paths, resolved defaults, and normalized validators. Configuration errors are detected here, before any form is created.
2 · ModelBuilder
Traverses the FieldNode[] and builds the initial model object, choosing the value by dataType: "" for string, 0 for number, false for boolean. Groups produce nested objects; arrays produce empty lists. The result is wrapped in signal().
3 · SchemaCompiler
Generates the schemaFn that form() will execute. The key technical insight: in JavaScript path["address"]["city"] is identical to path.address.city, which allows resolving dynamic paths at runtime using the same Signal Forms tree.
4 · Assembly
Calls form(model, schemaFn, {{ '{' }} injector {{ '}' }}) through the adapter layer. The resulting FieldTree is passed to the renderer; the (form, model) pair is exposed in the public API of <jf-form>.
{{ arch.engine }}
} @case ('dsl') {

Hybrid logic: DSL + registered functions

The library offers two complementary mechanisms for reactive conditions and custom validation. They are chosen based on the complexity of the logic and whether the JSON must be self-contained or may depend on registered TypeScript.

Expression DSL
A JavaScript-like expression evaluator without eval: the syntax tree is parsed once when the definition loads and re-evaluated with signals whenever the model changes. Can only read value and model; cannot call functions or access globals. Ideal when the JSON must be 100% self-contained.
Registered functions
When logic does not fit in the DSL — closures over Angular services, HTTP calls, complex calculations, access to injectables — it is registered in TypeScript with provideJsonForms() and referenced from the JSON by key. The context receives value(), valueAt(key), and model() as signals.
Rule: async always in registry
Async validation always goes through asyncValidators in the registry, never through the DSL. A validateAsync needs params, factory, onSuccess, and onError — TypeScript functions that are not serializable in JSON.
{{ arch.dsl }}
} @case ('renderer') {

Renderer and components

The render system is completely agnostic of the visual layer. The consumer registers their field components and the library instantiates and connects them to the Signal Forms FieldTree automatically.

FieldTypeRegistry
Maps each type value (string from the JSON) to the Angular component that renders it. The consumer fills the registry in provideJsonForms(). The same JSON definition renders with Material, PrimeNG, or plain HTML by changing only the registry.
FieldComponent contract
Each field component implements two inputs: field (the FieldTree node to bind [formField] and read reactive state: field().value(), field().errors(), field().touched()…) and config (the field configuration: label, props, type…).
Recursive FieldRenderer
Internal component that traverses the FieldNode[]. For each primitive node it dynamically instantiates the registry component (via NgComponentOutlet). For groups it recurses; for arrays it iterates over items with @for. The consumer never interacts with this component directly.
