Estructura de un campo

Cada entrada del array fields es un objeto con dos claves obligatorias (key y type) y el resto opcionales. El sistema infiere valores razonables cuando no se declaran explícitamente.

key requerido
Identificador único del campo en su nivel de anidación. Se convierte directamente en la clave del modelo resultante. En grupos, cada nivel tiene su propio espacio de nombres, por lo que la misma clave puede aparecer en distintos grupos sin conflicto.
type requerido
Controla qué control de UI se renderiza y cómo se serializa el valor. Ver pestaña Tipos de campo para la lista completa.
dataType
string · number · boolean. Se infiere del type cuando se omite (checkbox → boolean, number → number, el resto → string). Solo es necesario declararlo explícitamente cuando el tipo visual no coincide con el dato deseado.
label
Texto visible que acompaña al control en la UI. Si se omite, se muestra el valor de key auto-formateado (camelCase → palabras separadas).
defaultValue
Valor inicial del campo al crear o resetear el formulario. Debe ser compatible con dataType. Si se omite, los campos de texto parten de "", los numéricos de 0 y los booleanos de false.
props
Opciones adicionales del control:
  • placeholder — texto fantasma dentro del input mientras está vacío
  • hint — texto de ayuda que aparece debajo del campo
  • description — párrafo descriptivo que aparece encima del campo
  • options — array [{{ '{' }} value, label {{ '}' }}] obligatorio para select
hidden · disabled · readonly
Condiciones reactivas que se reevalúan en cada cambio del modelo. Acepta {{ '{' }} "expr": "..." {{ '}' }} para expresiones DSL self-contained, o {{ '{' }} "fn": "clave" {{ '}' }} para llamar a una función registrada en provideJsonForms(). Ver pestañas DSL y Funciones registradas.
validators · asyncValidators
Arrays de reglas de validación. Los síncronos se evalúan en cada cambio de valor y bloquean el envío si fallan. Los asíncronos soportan debounce para reducir llamadas al servidor. Ver pestaña Validadores.
fields solo group
Array de campos anidados que componen el grupo. En el modelo, el grupo genera un objeto cuya clave es la del campo padre y cuyo valor contiene los campos internos.
item solo array
Definición del elemento repetible. La UI genera botones «Añadir» y «Quitar» automáticamente. El modelo almacena un array de objetos según la estructura de item. Habitualmente item es un group con su propio fields.

Tipos de campo (type)

El type determina simultáneamente el control visual que se renderiza y el tipo de dato que el campo produce en el modelo. Los tipos primitivos generan un valor escalar; los compuestos (group, array) generan estructuras anidadas que se reflejan directamente en el modelo del formulario.

Validadores (validators)

Los validadores se declaran como array en la clave validators del campo. Todos admiten la clave opcional "message" para personalizar el texto de error. Hay cuatro categorías según su origen y modo de ejecución: estándar, DSL, síncrono registrado y asíncrono.

Estándar · self-contained
Reglas integradas en el motor que no requieren ningún registro externo: se declaran únicamente en el JSON. Son required (campo obligatorio), email (formato de correo), min / max (rango numérico), minLength / maxLength (longitud de texto) y pattern (expresión regular). Todos admiten "message" para sobrescribir el error por defecto.
expr · validación cross-field con DSL
El campo es válido cuando la expresión DSL evalúa a true. Útil para comparar el valor con otro campo del formulario, por ejemplo "value === model.emailConfirm". Se ejecuta en el mismo evaluador seguro del DSL (sin eval), por lo que también es self-contained. Ver pestaña DSL de expresiones para conocer la sintaxis disponible.
fn · función síncrona registrada
Referencia a una función registrada en provideJsonForms().validators. Recibe un contexto con value() (valor actual), valueAt(key) (valor de otro campo) y model() (snapshot completo del modelo). Debe devolver undefined si el campo es válido, o {{ '{' }} kind, message {{ '}' }} si no lo es. Ideal para lógica compleja que no cabe en el DSL: uso de servicios Angular, expresiones regulares avanzadas, etc.
Async · validación asíncrona
Se declara en el array asyncValidators del campo (no en validators). Referencia una clave registrada en provideJsonForms().asyncValidators. El campo "debounce" (ms) evita llamadas excesivas al servidor mientras el usuario escribe. El patrón usa resource de Angular: params define qué enviar, factory crea el resource, onSuccess interpreta el resultado y onError maneja fallos de red.

Grupos y arrays

Además de los tipos primitivos, puedes componer estructuras de datos complejas con group (objeto anidado) y array (lista repetible de elementos). Ambos se pueden combinar y anidar sin límite de profundidad.

group — objeto anidado

Agrupa campos relacionados bajo una clave común. El valor del grupo en el modelo es un objeto que contiene los valores de sus fields internos. Por ejemplo, un grupo "address" con campos "street" y "city" produce {{ '{' }} address: {{ '{' }} street: "...", city: "..." {{ '}' }} {{ '}' }} en el modelo.

Los grupos pueden tener su propio label, hidden, disabled y validadores, que afectan al bloque completo. Un grupo oculto también oculta todos sus campos internos.

array — lista repetible

Genera una lista de cero o más entradas donde el usuario puede añadir o eliminar filas. Cada fila sigue la definición de item, que habitualmente es un group con sus propios campos. La UI añade los botones «Añadir» y «Quitar» automáticamente.

El modelo almacena un array de objetos: si item.fields tiene "name" y "phone", el resultado es [{{ '{' }} name: "...", phone: "..." {{ '}' }}, ...]. La clave de item no aparece en el modelo; son las claves internas de fields las que se guardan directamente en cada posición del array.

Tip: los arrays pueden contener grupos que a su vez contengan otros arrays. La validación y las condiciones reactivas (hidden, disabled) funcionan a cualquier nivel de anidación.

DSL de expresiones

El DSL es un evaluador de expresiones tipo JavaScript integrado en el motor. Se usa en hidden, disabled, readonly y en validadores expr. No usa eval: el árbol sintáctico se parsea una vez al cargar la definición y se reevalúa de forma reactiva con señales cada vez que cambia el modelo, sin coste adicional.

Funciones y validadores registrados

Cuando la lógica no cabe en el DSL — closures sobre servicios Angular, llamadas HTTP, cálculos complejos, acceso a inyectables — se registra en TypeScript mediante provideJsonForms() en app.config.ts y se referencia desde el JSON por clave de texto. Hay tres registros independientes.

functions — condiciones reactivas
Funciones que controlan hidden, disabled y readonly. Reciben un contexto con:
  • value() — señal con el valor del campo actual
  • valueAt(key) — señal con el valor de otro campo por clave
  • model() — señal con el snapshot completo del modelo
Deben devolver true para activar la condición o false para desactivarla. Se referencian con {{ '{' }} "fn": "nombreClave" {{ '}' }} en el JSON. Se reevalúan reactivamente con señales Angular cada vez que cambia el modelo.
validators — validación síncrona
Validadores síncronos personalizados. Reciben el mismo contexto que las funciones. Deben devolver undefined si el campo es válido, o un objeto {{ '{' }} kind: string, message: string {{ '}' }} si no lo es. El kind identifica el tipo de error (útil para estilos condicionales) y message es el texto que ve el usuario. Se referencian con {{ '{' }} "kind": "fn", "fn": "clave" {{ '}' }} en el array validators.
asyncValidators — validación asíncrona
Validadores asíncronos basados en el primitivo resource de Angular. Cada entrada define cuatro propiedades:
  • params — función que extrae el valor a enviar (p.ej. el texto del campo)
  • factory — crea el resource con el loader asíncrono (fetch, HttpClient…)
  • onSuccess — interpreta el resultado y devuelve un error o undefined
  • onError — maneja fallos de red o servidor y devuelve un error de fallback
En el JSON se usan en el array asyncValidators con {{ '{' }} "kind": "clave", "debounce": 400 {{ '}' }}. El debounce retrasa la petición mientras el usuario sigue escribiendo, evitando llamadas innecesarias.

Visión general

Signal Forms está diseñada para estructuras tipadas en tiempo de compilación: el modelo y sus paths se conocen antes de ejecutar la app. signal-jsonforms invierte ese flujo: la forma del formulario llega en JSON en runtime. La librería actúa como intérprete — construye el modelo y la función de schema dinámicamente — sin reimplementar nada que Signal Forms ya ofrece.

Dirigida por datos
La estructura, validación y comportamiento del formulario se declaran en JSON. Modificar el formulario no requiere tocar TypeScript: basta con actualizar la definición.
Agnóstica de la capa visual
El núcleo no tiene dependencias de Material, PrimeNG ni ningún kit. La integración se hace vía FieldTypeRegistry: cambiar de kit de UI significa cambiar únicamente el registro, sin tocar el motor.
Fiel a Signal Forms
Estado, reactividad y validación los gestiona Signal Forms. La librería es un intérprete que traduce JSON → form(), no una reimplementación: delega siempre en la plataforma y aprovecha su reactividad nativa.
Aislada del riesgo experimental
@angular/forms/signals está marcada @experimental y puede cambiar entre versiones. Toda llamada a su API vive detrás de una única capa adaptadora: si la API rompe, se arregla en un solo módulo.
Tensión central: Signal Forms usa paths tipados estáticos (path.address.city). En runtime, la librería los resuelve dinámicamente con path["address"]["city"] — idéntico en JavaScript — sin tipado estático dentro del motor. El tipado se recupera opcionalmente en la frontera pública mediante genéricos.

Capas del sistema

Cuatro capas con fronteras claras. La regla de oro: solo la capa adaptadora importa @angular/forms/signals. El resto trabaja contra las interfaces internas de la librería. 

{{ arch.layers }}
Consumidor (app)
Define la configuración JSON del formulario y, opcionalmente, registra en provideJsonForms() los componentes de campo, validadores síncronos y asíncronos, y funciones para lógica condicional avanzada.
Capa de render
<jf-form> actúa como host. FieldRenderer recorre el FieldNode[] recursivamente e instancia dinámicamente el componente asociado a cada type, pasándole el nodo FieldTree y la configuración del campo.
Capa de motor
El núcleo puro: sin dependencias de UI ni de Signal Forms directamente. Comprende el Parser (JSON → IR), el ModelBuilder (IR → signal), el SchemaCompiler (IR → schemaFn) y los registros de expresiones, funciones y validadores.
Capa adaptadora
El único módulo que importa @angular/forms/signals. Envuelve form(), validate(), validateAsync(), applyWhen(), applyEach() y todos los validadores estándar detrás de interfaces propias de la librería.

Motor: de JSON a form()

El motor transforma la definición JSON en un formulario Signal Forms activo mediante cuatro pasos en cadena. El resultado es un par (model signal, FieldTree) que el renderer consume y que Signal Forms mantiene reactivo.

1 · Parser / Normalizador
Valida el JSON contra un meta-schema (Zod) y lo transforma en un array de FieldNode con paths absolutos calculados, defaults resueltos y validadores normalizados. Los errores de configuración se detectan aquí, antes de crear ningún formulario.
2 · ModelBuilder
Recorre el FieldNode[] y construye el objeto modelo inicial eligiendo el valor por dataType: "" para string, 0 para number, false para boolean. Los grupos producen objetos anidados; los arrays, listas vacías. El resultado se envuelve en signal().
3 · SchemaCompiler
Genera la schemaFn que form() ejecutará. La clave técnica: en JavaScript path["address"]["city"] es idéntico a path.address.city, lo que permite resolver paths dinámicos en runtime usando el mismo árbol de Signal Forms.
4 · Ensamblado
Llama a form(model, schemaFn, {{ '{' }} injector {{ '}' }}) a través de la capa adaptadora. El FieldTree resultante se pasa al renderer; el par (form, model) se expone en la API pública de <jf-form>.

Lógica híbrida: DSL + funciones registradas

La librería ofrece dos mecanismos complementarios para condiciones reactivas y validación personalizada. Se eligen según la complejidad de la lógica y si el JSON debe ser autocontenido o puede depender de TypeScript registrado.

DSL de expresiones
Un evaluador de expresiones tipo JavaScript sin eval: el árbol sintáctico se parsea una vez al cargar la definición y se reevalúa con señales cada vez que cambia el modelo. Solo puede leer value y model; no puede llamar funciones ni acceder a globales. Ideal cuando el JSON debe ser 100% self-contained.
Funciones registradas
Cuando la lógica no cabe en el DSL — closures sobre servicios Angular, llamadas HTTP, cálculos complejos, acceso a inyectables — se registra en TypeScript con provideJsonForms() y se referencia desde el JSON por clave. El contexto recibe value(), valueAt(key) y model() como señales.
Regla: async siempre en registro
La validación asíncrona siempre va por asyncValidators del registro, nunca por DSL. Un validateAsync necesita params, factory, onSuccess y onError — funciones TypeScript que no son serializables en JSON.

Render y componentes

El sistema de render es completamente agnóstico de la capa visual. El consumidor registra sus componentes de campo y la librería los instancia y conecta al FieldTree de Signal Forms automáticamente.

FieldTypeRegistry
Mapea cada valor de type (string del JSON) al componente Angular que lo renderiza. El consumidor rellena el registro en provideJsonForms(). La misma definición JSON renderiza con Material, PrimeNG o HTML plano cambiando solo el registro.
Contrato FieldComponent
Cada componente de campo implementa dos inputs: field (el nodo FieldTree para enlazar [formField] y leer estado reactivo: field().value(), field().errors(), field().touched()…) y config (la configuración del campo: label, props, type…).
FieldRenderer recursivo
Componente interno que recorre el FieldNode[]. Para cada nodo primitivo instancia dinámicamente el componente del registro (vía NgComponentOutlet). Para grupos recurse; para arrays itera sobre los items con @for. El consumidor nunca interactúa con este componente directamente.
